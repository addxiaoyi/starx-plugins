# StarX v0.2 Feature Design

## Overview

Extend StarX with player experience features, platform integrations, and utilities.
All new features follow the existing architecture: `Table → Repository → Service/Command → HTTP API`.

## 1. Database Schema (5 new tables)

### starx_punishments
- `id` VARCHAR(36) PK (UUID)
- `target_uuid` VARCHAR(36) NOT NULL
- `target_name` VARCHAR(16) NOT NULL
- `type` VARCHAR(16) NOT NULL (BAN / TEMPBAN / MUTE / KICK / WARN)
- `reason` VARCHAR(512)
- `staff_uuid` VARCHAR(36) NOT NULL
- `staff_name` VARCHAR(16) NOT NULL
- `created_at` BIGINT NOT NULL
- `expires_at` BIGINT (NULL = permanent)
- `active` BOOLEAN NOT NULL DEFAULT TRUE

### starx_staff_notes
- `id` VARCHAR(36) PK (UUID)
- `target_uuid` VARCHAR(36) NOT NULL
- `note` VARCHAR(1024) NOT NULL
- `severity` VARCHAR(16) NOT NULL (INFO / WARNING / CRITICAL)
- `staff_uuid` VARCHAR(36) NOT NULL
- `created_at` BIGINT NOT NULL

### starx_reports
- `id` VARCHAR(36) PK (UUID)
- `reporter_uuid` VARCHAR(36) NOT NULL
- `target_uuid` VARCHAR(36) NOT NULL
- `category` VARCHAR(32) NOT NULL (CHEATING / CHAT_ABUSE / SPAM / IGN.name / OTHER)
- `details` VARCHAR(512)
- `status` VARCHAR(16) NOT NULL DEFAULT 'PENDING' (PENDING / RESOLVED / DISMISSED)
- `resolved_by` VARCHAR(36)
- `resolved_at` BIGINT

### starx_announcements
- `id` VARCHAR(36) PK (UUID)
- `title` VARCHAR(128) NOT NULL
- `content` VARCHAR(2048) NOT NULL
- `created_by` VARCHAR(36) NOT NULL
- `created_at` BIGINT NOT NULL
- `expires_at` BIGINT

### starx_announcement_reads
- `announcement_id` VARCHAR(36) NOT NULL
- `player_uuid` VARCHAR(36) NOT NULL
- `read_at` BIGINT NOT NULL
- PK(`announcement_id`, `player_uuid`)

### starx_player_bindings
- `player_uuid` VARCHAR(36) PK
- `qq_id` VARCHAR(64)
- `discord_id` VARCHAR(64)
- `created_at` BIGINT NOT NULL

## 2. Repository Layer

All in `common/.../repository/`:

- `JdbcPunishmentRepository` — CRUD + query by player + by status
- `JdbcStaffNoteRepository` — CRUD + query by player
- `JdbcReportRepository` — CRUD + query by status/category + resolve
- `JdbcAnnouncementRepository` — CRUD + unread query + mark read
- `JdbcBindingRepository` — CRUD for player bindings

Raw JDBC with `DataSource` (same pattern as `JdbcUserRepository`).

## 3. HTTP API Endpoints (extend HttpApiServer)

All under `/api/v1/admin/`:

- `POST /punishments` — record punishment
- `GET /punishments?player=<uuid>` — query history
- `POST /notes` — add staff note
- `GET /notes?player=<uuid>` — query notes
- `POST /reports` — submit report
- `GET /reports?status=PENDING` — list reports
- `PUT /reports/:id/resolve` — resolve/dismiss report
- `POST /announcements` — publish announcement
- `GET /announcements?player=<uuid>` — get unread announcements
- `PUT /announcements/:id/read` — mark as read
- `POST /bindings/verify` — verify binding code

## 4. Velocity Commands

- `/report <player> <category>` — `starx.report`
- `/history <player>` — `starx.history`
- `/note <player> <content> [-s <INFO|WARNING|CRITICAL>]` — `starx.note`
- `/notes <player>` — `starx.note.list`
- `/announce <title> <content>` — `starx.announce`
- `/bind qq <code>` / `/bind discord` — `starx.bind`

## 5. Announcement Push

Velocity → Paper via Plugin Messaging channel `starx:announcement`.
Paper receives and shows ActionBar + Title popup to online players.
Also shown on login (fetched from DB via HTTP API or direct DB).

## 6. Integrations

### NapCat QQ Bot Module (replaces QqIntegrationModule)

```
NapCat (WS Server) ←→ StarX QqBotModule (WS Client)
NapCat (HTTP)      ←  StarX (HTTP POST sendMsg)
```

- Connect to NapCat Reverse WebSocket for receiving QQ events
- Send messages via NapCat HTTP API
- Handle `/bind qq` flow: generate code, send private msg via WS, verify
- Forward MC chat ↔ QQ group messages

### Discord Integration Module

- Discord Webhook for outgoing messages
- JDA for Slash Command handling
- `/list` / `/broadcast` commands via Discord → MC

### Vote (NuVotifier)

- Listen on NuVotifier port
- Record vote in database
- Execute configurable reward commands

### LuckPerms Context

- Register `ContextCalculator` for:
  - `server-type=limbo` (player in auth limbo)
  - `auth-status=authenticated|unauthenticated`
  - `maintenance=true|false`

## 7. Server Selector GUI (Phase 4)

- Server data fetched from `starx_servers` table (existing)
- Plugin Messaging `starx:gui` → Paper opens inventory GUI
- Click a server → Velocity switches player to that server

## 8. Vanish (Phase 4)

- Hook into Velocity TabList and PlayerListItem packets
- Remove vanished players from TabList/PlayerList
- Prevents vanished players from appearing in `/playerlist`
- LP context `vanished=true`
- Command: `/vanish` / `/v` — `starx.vanish`
